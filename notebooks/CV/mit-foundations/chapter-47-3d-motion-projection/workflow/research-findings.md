# Research findings — Textbook figure replication practice

We ran a deep-research pass (5 parallel search angles, 21 sources fetched, 25
falsifiable claims verified with 3-vote adversarial pass) to figure out what
the established practice is for reproducing textbook figures as runnable code.
The findings re-shaped this project's approach.

## TL;DR

**Use matplotlib. Don't try to match the book's Illustrator quality. Don't
switch tooling. Embedding the book's PNG alongside runnable code is academically
respectable.**

## The four findings that mattered

### 1. Major academic companion repos all use a minimal NumPy/SciPy/matplotlib + Jupyter stack

- Géron's [`handson-ml3`](https://github.com/ageron/handson-ml3) has a shared
  `save_fig(fig_id, tight_layout=True, fig_extension='png', resolution=300)`
  helper; `images/<topic>/README` explicitly says "Images generated by the
  notebooks" — code is the figure source, not a reverse-engineering target.
- `d2l-ai` (adopted at Stanford, MIT, Harvard, Cambridge — 500+ universities):
  "drafted in Jupyter notebooks, seamlessly integrating exposition figures,
  math, and interactive examples with self-contained code" — explicit hybrid
  pattern.
- `ctgk/PRML` (Bishop's PRML community notebooks, 94% Jupyter content): only
  uses Python 3 + NumPy + SciPy + Jupyter + matplotlib + scikit-learn. No TikZ,
  no Manim, no Plotly, no D3.

**Implication:** Stick with matplotlib. Adding TikZ / Manim / D3 introduces
tooling overhead without precedent in the projects this companion is most
similar to.

### 2. AI image-to-vector tools have matured but still require human editing

- DeTikZify (NeurIPS 2024 spotlight) "automatically synthesizes scientific
  figures as semantics-preserving TikZ graphics programs" but the
  write-compile-review loop is "cumbersome".
- VFig (March 2026) achieves VLM-Judge 0.829 on its own benchmark but
  **explicitly excludes math equations and natural images** from its scope.
- Classical tracers (Potrace, VTracer) produce unconstrained Bézier paths —
  unusable for editable diagrams (no concept of "axis", "arrow", or "label").
- Best multi-agent multimodal pipelines (PlotGen on MatPlotBench: 65.67%) still
  scored only 40.5% "Completely Accurate" in human evaluation.

**Implication:** AI vision is good as a first-draft tool but cannot autonomously
replicate technical figures. Human editing is still required — which is exactly
where the drag-and-drop editor approach steps in.

### 3. The "embed book PNG + runnable code that recomputes the math" pattern is academically respectable

- d2l-ai literally describes their notebooks as "exposition figures, math, and
  interactive examples with self-contained code" — the hybrid pattern.
- handson-ml3 embeds book PNGs for photos / architectural diagrams while
  generating data-driven plots from code.
- The Foundations of Computer Vision authors prescribe nothing about how
  reproductions should look or what tooling to use — they only say "we hope you
  will [reproduce the examples yourself]".

**Implication:** Pixel-matching the book is not required. Conceptually faithful
matplotlib + clearly explained math is the bar.

### 4. The Foundations of CV authors leave tooling open

The book's preface (verified directly): *"we learned a lot by writing and
working out the many examples we show in this book, and we hope you will too
by reading and reproducing the examples yourself."*

The book's official GitHub repo (Foundations-of-Computer-Vision/visionbook) is
a Quarto book-publishing pipeline only — no companion notebooks, no figure-
reproduction scripts, no prescribed tooling.

**Implication:** The implementer (this project's contributors) gets to choose
the approach. We chose: PyTorch (per the repo's hard requirement) + matplotlib
+ drag-and-drop editor for visual layouts.

## What we initially over-engineered (and corrected)

- **Tried to pixel-match the book's Illustrator art for Fig 47.1** through
  multiple matplotlib iterations of plane orientation and arrow placement.
  Wasted ~6 cycles before realizing the research-validated baseline (vanilla
  matplotlib, like Pantelis's own published work) was the right ceiling.
- **Considered switching to Manim** after a frustrating round on Fig 47.1.
  Verified manim isn't installed on host, the repo has no manim source code
  (Pantelis hosts compiled MP4s on a CDN we can't write to), and manim is for
  ANIMATIONS not static figures. Dropped the idea.
- **Considered using `kornia`** because it was mentioned in the kickoff brief.
  Verified Kornia was given as an *example* of an allowed library, not a
  requirement. Determined Kornia's actual API is better suited for image
  warping (Ch 38) than geometric diagrams (Ch 47). Stuck with raw PyTorch.

## What we landed on

| Layer | Choice | Why |
|---|---|---|
| Math | PyTorch | Required by the repo; sufficient for geometric/vector-field figures |
| Rendering | matplotlib (Rectangle, Circle, FancyArrowPatch, quiver, PathPatch) | The reference companion repos all do this; Pantelis's own published work uses it |
| Layout tuning | Browser drag-and-drop SVG editor (per-figure HTML, exports Python) | Drops per-iteration cost from minutes to seconds; produced the v6 of Fig 47.1 that the user confirmed was "much better" |
| Verification | Side-by-side comparison HTML (book PNG + my render) | Faster than eyeballing each figure individually; lets the user catch issues immediately |

## Open question we didn't resolve

**Copyright on embedding the book's PNGs in a public companion repo.** The
book is openly available at visionbook.mit.edu under a permissive license
(MIT Press), but specific figure rights may differ. We did NOT end up embedding
book PNGs in the published notebook — only used them as the editor's drag
target (which lives outside the repo, in `output/figure-editor/`). This sidesteps
the rights question entirely.
